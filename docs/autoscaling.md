# Autoscaling Runbook

This baseline uses KEDA to scale the HAPI FHIR deployment from Prometheus request-rate metrics with CPU as a secondary trigger. The autoscaler is intentionally capped by the PostgreSQL connection budget.

## Prerequisites

- KEDA `2.20.x` CRDs and controller are installed in the cluster.
- Kubernetes Metrics Server is installed so the KEDA CPU trigger can read pod CPU utilization.
- Prometheus can query HAPI Actuator metrics scraped from the ServiceMonitor.
- The Prometheus query endpoint is reachable from the KEDA operator.

If your Prometheus service is not `http://prometheus-operated.monitoring.svc.cluster.local:9090`, update `spec.triggers[]` entry `hapi-fhir-rps` in `manifests/autoscaling/hapi-fhir-scaledobject.yaml`.

## Apply

Install or upgrade the Helm release first, then apply the autoscaler:

```sh
helm upgrade --install hapi-fhir charts/hapi-fhir-deploy \
  --namespace fhir \
  --values charts/hapi-fhir-deploy/values.yaml
kubectl apply -f manifests/autoscaling/hapi-fhir-scaledobject.yaml
```

The `ScaledObject` targets `deploy/hapi-fhir-hapi-fhir-jpaserver` in namespace `fhir`.

## Scaling Policy

- Minimum replicas: `2`
- Maximum replicas: `5`
- Primary trigger: Prometheus request rate
- Secondary trigger: CPU utilization at `70%`
- Scale-down stabilization: `300` seconds

The Prometheus trigger uses this query:

```promql
sum(rate(http_server_requests_seconds_count{job="hapi-fhir-actuator"}[2m]))
```

KEDA requires the Prometheus scaler query to return a single scalar or one-element vector. The `sum(...)` wrapper turns the per-series Micrometer counter rates into one total request-rate signal for the HAPI FHIR deployment.

The current threshold is provisional:

```text
provisional_per_pod_rps_threshold = 5
```

This means the HPA generated by KEDA can scale toward:

```text
desired_replicas = ceil(total_hapi_fhir_rps / 5)
```

Replace `threshold: "5"` only after load testing identifies a sustained per-pod RPS target that preserves latency, error rate, CPU headroom, and PostgreSQL connection headroom.

## Connection Budget

The committed starter environment assumes:

```text
postgres_max_connections = 100
reserved_connections = 50
hikari_maximum_pool_size = 10
```

The autoscaling ceiling is:

```text
maxReplicas <= floor((postgres_max_connections - reserved_connections) / hikari_maximum_pool_size)
maxReplicas <= floor((100 - 50) / 10)
maxReplicas <= 5
```

At the committed `maxReplicaCount: 5`, HAPI FHIR can open at most:

```text
max_app_connections = maxReplicaCount * hikari_maximum_pool_size
max_app_connections = 5 * 10 = 50
```

That leaves `50` PostgreSQL connections reserved for admin access, migrations, monitoring, maintenance, and provider overhead.

Do not raise `maxReplicaCount`, `spring.datasource.hikari.maximumPoolSize`, or the KEDA RPS threshold without recalculating this equation against the target database's real `max_connections`.

The runtime rollout workstream keeps Hikari at `10` until load tests justify a different value. Raising Hikari to `20` would reduce this native PostgreSQL ceiling to `2` replicas unless PostgreSQL capacity changes or PgBouncer transaction pooling is introduced.

## PgBouncer Requirement

PgBouncer transaction pooling is required before native PostgreSQL budget is exceeded. Introduce PgBouncer and recompute `maxReplicaCount` when either of these is true:

- Required replicas are greater than `floor((max_connections - reserved) / hikari_maximum_pool_size)`.
- Required per-pod Hikari pool size would consume the reserved PostgreSQL connection budget.

After adding PgBouncer, document the new pooler sizing, server connection cap, transaction pooling mode, health checks, rollback path, and how application session features are validated under transaction pooling.

## Verification

Check KEDA and the generated HPA:

```sh
kubectl -n fhir get scaledobject hapi-fhir-jpaserver
kubectl -n fhir describe scaledobject hapi-fhir-jpaserver
kubectl -n fhir get hpa hapi-fhir-jpaserver
```

Confirm Prometheus can evaluate the request-rate signal:

```promql
sum(rate(http_server_requests_seconds_count{job="hapi-fhir-actuator"}[2m]))
```

Generate test traffic, then watch desired replicas stay within the connection-budget cap:

```sh
kubectl -n fhir get hpa hapi-fhir-jpaserver --watch
kubectl -n fhir get deploy hapi-fhir-hapi-fhir-jpaserver --watch
```

If scaling does not activate, inspect:

```sh
kubectl -n fhir describe scaledobject hapi-fhir-jpaserver
kubectl -n keda logs deploy/keda-operator
```

Common causes are a different Prometheus service address, missing Metrics Server, missing `job="hapi-fhir-actuator"` labels, or ServiceMonitor selector filters that prevent scraping.

## Rollback

Remove the autoscaler and return the deployment to the baseline replica count:

```sh
kubectl delete -f manifests/autoscaling/hapi-fhir-scaledobject.yaml
kubectl -n fhir scale deploy/hapi-fhir-hapi-fhir-jpaserver --replicas=2
```

If the Helm release also changed, roll it back separately:

```sh
helm -n fhir rollback hapi-fhir
```
